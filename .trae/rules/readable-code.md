命名约定：
    使用有意义的、描述性的名称。
    遵循项目或语言的命名规范。
    避免使用下划线开头的名称（除非是约定俗成的，如私有变量）。
    避免使用下划线结尾的名称（除非是约定俗成的，如 C++ 中的 `_t` 类型后缀）。
    避免使用混合大小写的名称（除非是约定俗成的，如 Java 中的 `camelCase`）。
    避免使用单字母名称（除非是约定俗成的，如循环中的 `i`）。
    避免使用描述性与功能相关的名称（如 `add`、`remove` 等）。
    避免使用描述性与上下文相关的名称（如 `user`、`customer` 等）。
    避免使用描述性不强的名称（如 `temp`、`tmp` 等）。
    避免使用描述性过强的名称（如 `maxValue`、`minValue` 等）。
    避免使用描述性与实现细节相关的名称（如 `calculateTotal`、`processData` 等）。
代码组织：
    相关代码放在一起。
    函数只做一件事。
    避免使用过长的函数。
    函数应该是小而专注的。
    避免使用全局变量。
    避免使用深层嵌套的代码。
    避免使用复杂的控制结构。
    保持适当的抽象层次。
注释与文档：
    注释应该解释为什么，而不是做什么。
    为公共 API 提供清晰的文档。
    更新注释以反映代码变化。
    注释应该解释代码的意图和实现细节。
    注释应该避免使用代码中的细节，而应该使用更高级别的概念。
    注释应该是简洁的、直接的。
    避免使用无意义的注释。
    注释应该避免使用专业术语，除非是约定俗成的。
    注释应该避免使用复杂的词汇或术语。
    注释应该是代码的补充，而不是重复代码。
