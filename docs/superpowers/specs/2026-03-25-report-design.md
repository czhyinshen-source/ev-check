# 报告生成模块设计方案

**项目：** EV Check System（运行环境检查系统）
**日期：** 2026-03-25
**状态：** 已确认

---

## 一、需求概述

报告生成模块为检查执行结果提供网页展示和 PDF 导出功能。

- **输出格式：** 网页展示 + PDF 下载（无需 Excel）
- **报告内容：** 详细报告（概览 + 执行结果汇总 + 各检查项详情含原始值/期望值/诊断信息 + 失败项高亮）
- **交互功能：** 增强交互（筛选/排序/分页 + 展开/折叠详情 + 跳转对比历史快照）
- **PDF 导出：** 整体导出当前报告为 PDF
- **入口：** 检查执行后可跳转查看 + 历史报告列表回溯

---

## 二、技术方案

**架构模式：** 轻量前端渲染

**页面结构：**
- `/reports` — 历史报告列表页
- `/reports/{result_id}` — 报告详情页

**技术选型：**
- PDF 库：`html2pdf.js`（浏览器端直接生成，无需后端）
- 无需新增后端 API（复用现有检查结果 API）
- 无需新增数据库表（复用现有 `CheckResult` / `CheckResultDetail` 数据）

**页面入口：**
1. 检查执行完成 → 自动跳转 `/reports/{result_id}` 查看报告
2. 历史报告列表 → 用户从 `/reports` 选择任意历史报告查看

**三层页面结构：**
```
报告列表页 (/reports)
    └── 报告详情页 (/reports/{result_id})
            ├── 概览区（执行信息、总览统计）
            ├── 汇总表（各检查项通过率，缩略展示）
            ├── 明细区（展开后：每台服务器的原始值/期望值/状态）
            └── PDF 导出按钮
```

## 三、报告列表页设计

每条历史报告记录以卡片形式展示：

```
┌─────────────────────────────────────────────────────────┐
│ ● 检查规则: 生产环境日常检查              执行时间: 2026-03-25 14:30 │
│   关联快照: 2026-03-20 快照                               │
│   触发方式: 手动执行          耗时: 2分35秒               │
│   ─────────────────────────────────────────────────────── │
│   检查项: 48  │  ✅ 通过 42  │  ❌ 失败 5  │  ⚠️ 异常 1   │
│                                           [查看报告] [导出PDF] │
└─────────────────────────────────────────────────────────┘
```

**列表筛选（顶部）：**
- 按执行时间范围筛选
- 按检查规则筛选
- 按状态筛选（全部 / 含失败 / 含异常）

**列表排序：** 默认按执行时间倒序（最新在上）

## 四、报告详情页设计

### 4.1 概览区（顶部卡片）

| 字段 | 内容示例 |
|------|---------|
| 执行ID | #12345 |
| 执行时间 | 2026-03-25 14:30:22 |
| 触发方式 | 手动执行 / 定时任务 |
| 执行耗时 | 2分35秒 |
| 关联规则 | "生产环境日常检查" |
| 关联快照 | "2026-03-20 快照" |

### 4.2 汇总统计

四个核心指标卡片横向排列：

| 指标 | 示例 |
|------|------|
| 总检查项数 | 48 |
| 通过 | 42 (87.5%) ✅ |
| 失败 | 5 (10.4%) ❌ |
| 异常 | 1 (2.1%) ⚠️ |

每个指标卡片点击可筛选对应状态的检查项明细。

### 4.3 汇总表（缩略展示）

每个检查项默认以一行缩略形式展示：

```
┌─────────────────────────────────────────────────────────┐
│ ▼ /etc/hosts 文件检查         类型: 文件系统   服务器: 3   │
│   汇总: ✅ 3/3 通过   ❌ 0 失败   ⚠️ 0 异常              │
└─────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────┐
│ ▼ nginx 进程检查              类型: 进程       服务器: 3   │
│   汇总: ✅ 2/3 通过   ❌ 1 失败   ⚠️ 0 异常              │
│   ❌ DB服务器-01 (192.168.1.20) - 进程不存在             │
└─────────────────────────────────────────────────────────┘
```

**缩略视图交互：**
- 默认收起，只显示汇总行
- 点击展开按钮（▼）展开该检查项的服务器明细表
- 失败的检查项在缩略视图中直接显示「❌ 服务器名 - 简短失败原因」
- 页面顶部提供「全部展开 / 全部折叠」快捷按钮

**失败项高亮：**
- 表格行背景：失败 = 浅红色高亮，异常 = 浅黄色高亮，通过 = 无特殊样式
- 汇总表默认按「失败 > 异常 > 通过」排序，让问题优先呈现

### 4.4 汇总表筛选与排序

- 按「检查类型」筛选
- 按「结果状态」筛选（全部 / 通过 / 失败 / 异常）
- 支持按各列排序（检查项名称、类型、服务器数、通过数、失败数、异常数）

### 4.5 明细区（展开视图）

点击「查看详情」，展开该检查项在所有服务器上的详细结果：

| 服务器 | IP | 检查结果 | 期望值 | 实际值 | 详情 |
|-------|-----|---------|-------|-------|------|
| Web服务器-01 | 192.168.1.10 | ✅ 通过 | 文件存在，权限 644 | 权限 644 | — |
| DB服务器-01 | 192.168.1.20 | ❌ 失败 | 权限 644 | 权限 777 | 权限不符合要求 |
| App服务器-01 | 192.168.1.30 | ✅ 通过 | 文件存在，权限 644 | 权限 644 | — |

明细区同样支持筛选和排序。

## 五、PDF 导出设计

- 使用 `html2pdf.js` 在浏览器端直接生成 PDF
- 导出按钮位于报告详情页顶部
- PDF 内容包含当前报告的完整信息（概览 + 汇总 + 明细）
- PDF 样式：深交所风格，与 Web 页面视觉一致
- 一次性生成完整报告，无需选择性导出

## 六、实现要点

### 6.1 前端改动

1. **新增报告列表页** `app/static/reports.html`
2. **改造报告详情页** — 整合到 dashboard 或新建独立页面
3. **新增报告相关 JS** `app/static/js/reports.js` — 负责数据拉取、渲染、筛选排序逻辑
4. **引入 `html2pdf.js`** — PDF 生成
5. **复用现有 API：**
   - `GET /api/checks/` — 历史检查结果列表
   - `GET /api/checks/{id}` — 单次检查执行详情
   - `GET /api/check-rules/` — 关联规则信息

### 6.2 后端改动

- 无需新增 API（复用现有 API）
- 可选：根据展示需求适当扩展 `CheckResult` 返回字段（如增加执行耗时汇总、关联快照名称等）

### 6.3 页面跳转逻辑

- 检查执行完成后，前端自动跳转到 `/reports/{result_id}`
- 报告列表页的「查看报告」按钮跳转到报告详情页
- 报告列表页的「导出PDF」按钮直接触发当前卡片对应报告的 PDF 下载

---

## 七、非功能性要求

- 页面加载时显示骨架屏或加载状态
- 明细展开/折叠使用平滑动画过渡
- PDF 导出时显示 loading 提示，导出完成后自动下载
- 响应式布局，支持在不同屏幕尺寸下查看报告
- 历史报告数据来自数据库持久化，无需额外缓存
